# Phase 1: Kernel Implementation

## Scope

Phase 1 delivers the minimal kernel: a single long-running process (`studio-orchestrator`) that accepts bundle submissions, drives a linear DAG of worker tasks, and writes outcomes to SQLite. The kernel is the foundation on which Phases 2+ add the bundler agent, pre-execution review tracks, approval matrix, MCP surface, and mid-flight steering machinery.

Phase 1 is scoped to the following components from the v1.1 spec:
- Architecture: orchestrator core process, SQLite schema, crash recovery (kill-all)
- Worker runner: `LocalBwrapWorkerRunner` with the `permissive` network isolation escape hatch
- Worker RPC protocol: all 14 methods defined; stubs for deferred methods
- Capability manifest: YAML schema validated on submission, used for RPC dispatch checks
- Task DAG schema: linear DAGs only (no branches, no aggregation, no expansion)
- DAG executor: node lifecycle (`pending ‚Üí ready ‚Üí running ‚Üí completed/failed`), FIFO scheduling, heartbeat liveness
- Bundle state machine: 12 enum values defined, 8 transition handlers implemented (see below)
- CLI surface: `studio submit`, `studio approve`, `studio reject`, `studio list`, `studio show`, `studio show-worker`, `studio kill`, `studio status`

Phase 1 explicitly excludes: the bundler agent, pre-execution review tracks (adversarial, security, QA), the approval matrix evaluator, MCP and GitHub Issues surfaces, mid-flight steering (Pause/Redirect/Abort/Rollback), dynamic expansion, aggregator nodes, gate nodes, and the artifact protocol layer. Worker-to-worker artifact transfer uses the git worktree instead.

## 1. Kernel submission format

Bundles are submitted as JSON files via `studio submit <path>`. The file must conform to this schema:

```yaml
# Phase 1 submission schema
submission:
  schema_version: "1.0-phase-1"

  bundle_input:
    idea: str                         # free-text description, 1-65536 bytes
    filed_by: str                     # identity string (default: "reviewer")
    filed_at: str                     # ISO8601, auto-populated by kernel if missing

  capability_manifest:
    schema_version: "1.0"
    subject:
      kind: "bundle"
      id: "<auto-generated on submission>"
    grants:
      filesystem: { reads: [...], writes: [...], working_tree: { ... } }
      network: { egress: [...], ingress: { enabled: false }, dns: { enabled: true|false } }
      process: { exec: [...], spawn_subtasks: { enabled: false } }
      secrets: [...]
      rpc: { methods: [...], artifact_access: { reads: [...], writes: [...] } }
      resources: { ... }
    metadata:
      rationale: "<justification>"
      requested_by: "<filed_by value>"
      expires_at: null

  task_dag:
    schema_version: "1.0"
    bundle_id: "<auto-generated on submission>"
    bundle_manifest_ref: "<reference to manifest>"

    nodes:
      - id: "<task-id>"
        kind: worker                   # only 'worker' kind in Phase 1
        task_manifest_ref: "<pointer to capability manifest subset>"
        spec:
          objective: "<natural-language description>"
          inputs: { artifacts: [], params: {} }
          outputs: { artifacts: [] }
          success_criteria:
            - kind: exit_code
              expected: 0
          retry_policy:
            max_attempts: 1
            backoff: "immediate"

    edges:
      - from: "<node id>"
        to: "<node id>"
        condition:
          kind: on_success             # only 'on_success' in Phase 1

    entry_nodes: ["<first node id>"]
    exit_nodes: ["<last node id>"]

    expansion_policy:
      allow_dynamic_expansion: false
      max_total_nodes: 0
      max_depth: 0

    metadata:
      created_by: "<filed_by value>"
      created_at: "<auto-populated>"
      rationale: "<bundle-level justification>"
```

**Validation rules at submission:**
1. `task_dag.nodes` forms a linear chain (each node has at most one predecessor and one successor via `dag_edges`).
2. No gate or aggregator nodes.
3. `expansion_policy.allow_dynamic_expansion` is `false`.
4. `retry_policy.max_attempts` is 1 (no retry in Phase 1).
5. The `capability_manifest` passes schema validation.
6. Every node's `task_manifest_ref` resolves to a manifest that is a subset of the bundle manifest (using the `is_subset()` algorithm from the spec).

The kernel auto-populates `bundle_id` (ULID), `filed_at`, `subject.id`, and `task_dag.bundle_id`. On successful validation, the kernel creates a `bundles` row in `PROPOSED` state and returns the bundle ID to the CLI.

## 2. Bundle state machine (Phase 1 subset)

Define all 12 `BundleState` enum values from the spec. Phase 1 implements transition handlers for these 9 transitions:

| # | From | Trigger | To | Actor | Side effects | RPC notifications |
|---|------|---------|----|-------|-------------|-------------------|
| 1 | (none) | `bundle_input_received` | `PROPOSED` | Orchestrator | INSERT `bundles`, INSERT `dag_nodes`, INSERT `dag_edges`, INSERT `audit_log` | (none) |
| 1a | `PROPOSED` | `kernel_direct_approval` | `APPROVED` | Reviewer | UPDATE `bundles.state`, UPDATE `bundles.approved_at`, UPDATE `bundles.approved_by`, INSERT `approval_decisions`, INSERT `audit_log` | (none) |
| 1b | `PROPOSED` | `kernel_direct_rejection` | `REJECTED` | Reviewer | UPDATE `bundles.state`, UPDATE `bundles.completed_at`, UPDATE `bundles.outcome_json`, INSERT `approval_decisions`, INSERT `audit_log` | (none) |
| 6 | `APPROVED` | `execution_started` | `IN_PROGRESS` | Orchestrator | UPDATE `bundles.state`, INSERT `audit_log` | (none) |
| 9 | `IN_PROGRESS` | `all_exit_nodes_terminal` | `VERIFYING` | Orchestrator | UPDATE `bundles.state`, INSERT `audit_log` | (none) |
| 17 | `VERIFYING` | `verification_passed` | `COMPLETE` | Orchestrator | UPDATE `bundles.state`, UPDATE `bundles.completed_at`, UPDATE `bundles.outcome_json`, INSERT `audit_log` | (none) |
| 19 | `VERIFYING` | `verification_failed_no_rollback` | `FAILED` | Orchestrator | UPDATE `bundles.state`, UPDATE `bundles.completed_at`, UPDATE `bundles.outcome_json`, INSERT `audit_log` | (none) |
| 25 | `IN_PROGRESS` | `bundle_failed_during_execution` | `FAILED` | Orchestrator | UPDATE `bundles.state`, UPDATE `bundles.completed_at`, UPDATE `bundles.outcome_json`, INSERT `audit_log` | (none) |

Transition 1a (`PROPOSED ‚Üí APPROVED`) is marked `[PHASE-1-ONLY]`. It exists only when `kernel_mode = true` (set at orchestrator startup). In Phase 2 (full system mode), this transition is removed and `bundler_completed` (transition 2) is the only legal path out of `PROPOSED`.

**Transition 1b** (`PROPOSED ‚Üí REJECTED`) [PHASE-1-ONLY]: Mirrors transition 1a. Allows the reviewer to reject a submitted bundle directly from the CLI (`studio reject <id>`) without entering the review flow. Legal only when `kernel_mode = true`. Removed in Phase 2 when the bundler and approval matrix exist.

The remaining 4 enum values (`IN_REVIEW`, `PAUSED`, `REDIRECTING`, `PARKED`) are defined in the enum but never reached in Phase 1 execution. Their presence prevents a schema migration in Phase 2.

**Illegal transitions.** Any attempt to execute a transition not in the table above raises `IllegalTransitionError`. The error serializes as a JSON-RPC error with code `-32001`.

**Surface observability.** Phase 1 uses SQLite as the sole state surface. `bundles.state` is updated atomically with the transition. CLI `studio show <id>` reads state directly.

## 4. DAG executor (Phase 1 subset)

### Node lifecycle

Phase 1 implements the linear subset: `pending ‚Üí ready ‚Üí running ‚Üí completed|failed`. Since all Phase 1 DAGs are linear and all edges are `on_success`, the homogeneity and aggregator rules are not exercised.

Entry nodes (no incoming edges) are born `ready`. For all other nodes, `pending ‚Üí ready` fires when the single predecessor reaches `completed`.

`ready ‚Üí running` is gated on the global worker semaphore (cap 4). Phase 1 uses FIFO scheduling by `ready_at` timestamp.

`running ‚Üí completed` fires on `worker.final_report` with `outcome: success`. `running ‚Üí failed` fires on `outcome: failure`, process exit (non-zero), or heartbeat timeout.

`failed` on any node in a linear DAG fails the bundle (transition 25). There is no retry in Phase 1.

### Heartbeat-driven liveness

On every `worker.heartbeat` notification: update `workers.last_heartbeat` to `now()`. On the first heartbeat from a worker in `pending` state: additionally transition `workers.state` from `pending ‚Üí running`. Subsequent heartbeats from a worker in `running` state update only `last_heartbeat`.

A worker that exceeds 2x its expected timeout (from `resources.wall_time_limit` in its task manifest) without a heartbeat is considered wedged. The executor kills the worker process (SIGTERM, then SIGKILL after 10 seconds) and transitions the node to `failed` with `failure_reason = 'heartbeat_timeout'`.

### Execution start

When bundle state transitions `APPROVED ‚Üí IN_PROGRESS` (transition 6), the executor:
1. Reads `dag_nodes` and `dag_edges` for the bundle.
2. Sets the entry node(s) to `ready`.
3. Computes the ready set and dispatches.

## 5. RPC dispatcher

Implement all 14 methods from the Worker RPC protocol section. The following are implemented in full:

- `worker.heartbeat` ‚Äî notification handler per the heartbeat specification above
- `worker.log` ‚Äî writes structured log entries to `audit_log`
- `worker.progress_report` ‚Äî updates `dag_nodes.output_json` with progress data
- `worker.final_report` ‚Äî terminal call; triggers node state transition
- `worker.pause` ‚Äî returns `-32000 method_not_implemented` in Phase 1
- `worker.resume` ‚Äî returns `-32000 method_not_implemented` in Phase 1
- `worker.cancel` ‚Äî returns `-32000 method_not_implemented` in Phase 1
- `worker.query_status` ‚Äî returns current `dag_nodes.state` and `workers.last_heartbeat`
- `worker.inject_context` ‚Äî returns `-32000 method_not_implemented` in Phase 1
- `worker.prepare_handoff` ‚Äî returns `-32000 method_not_implemented` (superseded)
- `cap.request` ‚Äî returns `-32000 method_not_implemented` in Phase 1
- `cap.check` ‚Äî full implementation using `op_descriptor` structured format
- `artifact.request` ‚Äî returns `-32000 method_not_implemented` (superseded)
- `worker.request_human_input` ‚Äî returns `-32000 method_not_implemented` in Phase 1

### cap.check implementation

`op_descriptor` format: `<category>.<operation>[:<resource>]`

The dispatcher parses the category prefix and dispatches:

- `filesystem.read:<path>` or `filesystem.write:<path>` ‚Äî checks worker's `filesystem` grants for path containment with recursive flag handling
- `network.egress:<host>:<port>` or `network.egress:<cidr>` ‚Äî checks worker's `network.egress` grants for destination containment
- `process.exec:<binary_path>` ‚Äî checks worker's `process.exec` grants for exact binary path match
- `rpc.method:<method_name>` ‚Äî checks worker's `rpc.methods` grants via wildcard matching (`"artifact.*"` covers `"artifact.publish"`)
- `rpc.artifact_access.read:<descriptor>` or `rpc.artifact_access.write:<descriptor>` ‚Äî checks worker's `rpc.artifact_access` grants via glob matching on the descriptor fields
- `secrets.fetch:<name>` ‚Äî checks worker's `secrets` grants for name match

Returns `{allowed: true, capability_id: "<ulid>"}` on match, `{allowed: false, capability_id: null}` on no match.

The `cap.check` handler is a pure function. It performs no I/O beyond reading the worker's manifest (cached at connection setup).

### RPC dispatcher capability check

Every method call is validated against the worker's `rpc.methods` grant. The worker's manifest `rpc.methods` list is checked for a pattern covering the called method. `"worker.*"` covers `worker.heartbeat` and `worker.final_report`. If no pattern matches, the dispatcher returns `-32001 capability_denied`.

## 6. Worker runner

Use `LocalBwrapWorkerRunner` with bubblewrap isolation. Phase 1 supports a `kernel.network_isolation` setting in `settings.json`:

```jsonc
{
  "kernel": {
    "mode": true,                    // enables kernel-mode (transition 1a and CLI-only surface)
    "network_isolation": "permissive" // Phase 1 only: allows all egress, bypasses the host-side proxy
  }
}
```

`"permissive"` allows all network egress from worker containers. This is a Phase 1-only escape hatch that exists because the host-side egress proxy (hostname-based enforcement) is deferred. In Phase 2, `network_isolation` defaults to `"enforcing"` and the `permissive` value is removed.

Worker spawn sequence:
1. WorkerRunner allocates a worktree on a sub-branch `bundle/<bundle-id>/worker-<n>`.
2. Capability manifest is translated to bwrap flags (filesystem ro/rw mounts, `--unshare-net` for network isolation, `--seccomp` for syscall filtering).
3. Worker process is spawned with `STUDIO_WORKER_TOKEN` in its environment.
4. Worker connects back over the Unix domain socket, presents token, and is bound.
5. Orchestrator delivers the task spec via the RPC channel.

## 7. Developer worker (Phase 1 stub)

The Phase 1 developer worker is an OpenCode invocation with a fixed prompt constructed from the task spec. The worker:

1. Connects to the orchestrator via RPC using `STUDIO_WORKER_TOKEN`.
2. Sends `worker.heartbeat` with `phase: "starting"` immediately on connection.
3. Reads the task spec (delivered over RPC after connection binding).
4. Constructs a prompt: the `objective` field from the node spec, plus the file context (`inputs.artifacts` paths resolved to worktree paths).
5. Invokes OpenCode CLI with the prompt.
6. During execution: sends `worker.heartbeat` with appropriate phase on each state transition, maximum 60-minute idle interval.
7. On OpenCode completion: calls `worker.final_report` with outcome, files changed, and summary.
8. Commits to its sub-branch and exits.

The worker does not call `cap.check`, `cap.request`, `artifact.publish`, `artifact.fetch`, or any other RPC method in Phase 1. All inter-worker data flow uses the git worktree (each worker's output is committed to its branch; downstream workers merge the predecessor branch before starting).

## 8. Crash recovery

Implement the 6-step reconciliation protocol from the Architecture section. Phase 1 notes:

- **Step 1:** "Scan workers for rows in state `running` or `paused`" ‚Äî the `paused` branch is a no-op in Phase 1 (no code path produces paused workers). The agent writes the logic; it is never exercised in Phase 1.
- **Step 4:** "Replay unread approval decisions" ‚Äî the `approval_requests` table is unused in Phase 1 (no MCP, no GitHub Issues). This step is a no-op. The agent writes the logic; the SELECT returns zero rows in Phase 1.
- **Step 5:** Bundles in `verifying` re-trigger the verification check. In Phase 1, verification is a no-op (the kernel transitions `verifying ‚Üí complete` when all workers finish). After crash, bundles in `verifying` are re-ticked: the executor recomputes the ready set and the bundle proceeds normally.

Reconciliation is idempotent. Running it twice on the same SQLite state produces the same result.

## 9. CLI surface

Eight commands:

**`studio submit <path>`**
Reads the submission JSON, validates it, creates a `bundles` row in `PROPOSED` state, and prints the bundle ID.
```
$ studio submit tests/fixtures/hello-world-bundle.json
Bundle submitted: 01JXYZ...
```

**`studio approve <bundle-id>`**
Executes transition 1a (`PROPOSED ‚Üí APPROVED`). Legal only when `kernel_mode = true` and bundle is in `PROPOSED` state. On success, the kernel executes transition 6 (`APPROVED ‚Üí IN_PROGRESS`) immediately and begins execution.
```
$ studio approve 01JXYZ...
Bundle 01JXYZ... approved. Starting execution.
```

**`studio reject <bundle-id>`**
Executes transition 1b (`PROPOSED ‚Üí REJECTED`). Legal only when `kernel_mode = true` and bundle is in `PROPOSED` state. Prompts for an optional reason string; if none given, records `"rejected via CLI"`.
```
$ studio reject 01JXYZ...
Bundle 01JXYZ... rejected.
```

**`studio list`**
Lists all non-terminal bundles with state, age, and first line of idea. Sorted by `created_at` descending. Accepts `--state <state>` to filter and `--json` for machine-readable output.
```
$ studio list
ID              STATE        AGE     IDEA
01JXYZ...       IN_PROGRESS  4m      Build a hello-world web server
01JABC...       PROPOSED     12m     Add logging to API
```

**`studio show <bundle-id>`**
Reads `bundles` and `dag_nodes` for the bundle and prints a status report:
```
$ studio show 01JXYZ...
Bundle: 01JXYZ...
State: IN_PROGRESS
Idea: Build a hello-world web server
Nodes: 3 total, 1 completed, 1 running, 1 pending
```

**`studio show-worker <worker-id>`**
Shows worker detail: state, current phase from last heartbeat, time since last heartbeat, tail of last 20 log lines from `audit_log` for this worker, exit reason if terminal.
```
$ studio show-worker w_01JXYZ...
Worker:       w_01JXYZ...
Bundle:       01JXYZ...
State:        running
Phase:        writing-code
Last hb:      23s ago
Log (last 20 lines):
  [info] Wrote src/main.py
  [info] Running tests...
```

**`studio kill <bundle-id>`**
Aborts an in-progress bundle. Sends SIGTERM to all running worker processes, waits up to 30 seconds for clean exit, then SIGKILL. Transitions `workers.state` to `killed`. Transitions `bundles.state` to `failed` with `exit_reason = "killed_by_reviewer"`. Legal from `IN_PROGRESS` only.
```
$ studio kill 01JXYZ...
Sending SIGTERM to 1 worker(s)...
Worker w_01JXYZ... killed.
Bundle 01JXYZ... failed.
```

**`studio status`**
Lists all non-terminal bundles:
```
$ studio status
01JXYZ...  IN_PROGRESS  Build a hello-world web server
01JABC...  PROPOSED     Add logging to API
```

## 13. Deferred to Phase 2

The following are explicitly deferred and must not be implemented in Phase 1:

- Bundler agent (planning, scoring, proposal generation)
- Pre-execution review tracks (adversarial critique, security review, QA verification planning)
